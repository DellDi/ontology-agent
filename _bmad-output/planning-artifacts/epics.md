# ontology-agent - Epic Breakdown

## 概览

本文档提供 `ontology-agent` 的完整 Epic 与 Story 拆解，将 PRD 与架构文档中的需求转换为可实施的故事清单，可直接作为后续开发与实施排期的输入。

## 需求清单

### 功能需求

FR1: 用户可以通过自然语言输入与物业分析相关的问题，系统应接受完整问题句而不要求用户掌握 SQL、报表路径或字段名。  
FR2: 系统应将用户问题识别为查询、趋势分析、对比分析、归因分析或后续决策支持等分析类型，并输出可用于后续执行的结构化意图结果。  
FR3: 系统应从用户问题中抽取目标指标、实体对象、时间范围、比较方式和约束条件，并允许这些抽取结果在会话中被用户纠正。  
FR4: 当问题属于归因分析或影响分析时，系统应基于业务实体关系扩展候选影响因素，并将这些因素纳入分析上下文。  
FR5: 系统应为复杂分析问题生成由多个步骤组成的分析计划，并向用户展示计划步骤、依赖关系或执行顺序。  
FR6: 系统应按分析计划调用所需工具或数据能力执行各个步骤，且单次分析至少支持多步执行而不是仅支持一次查询。  
FR7: 系统应在分析执行过程中向用户反馈当前步骤、阶段结果、关键发现或执行状态，而不是仅在分析全部结束后返回最终答案。  
FR8: 系统应对归因分析输出排序后的原因列表，并为每个原因提供可供用户理解的证据摘要。  
FR9: 用户可以在初始结论基础上继续追问、补充因素、缩小范围或纠正方向，系统应在保留会话上下文的前提下触发下一轮分析循环。  
FR10: 当用户发现分析方向偏差时，系统应允许用户对目标指标、分析范围、候选因素或步骤计划进行纠正，并重生成后续计划。  
FR11: 系统应保存分析会话的核心输入、计划、主要步骤结果和最终结论，以便用户回看和复盘。  
FR12: 系统只能在用户被授权的数据范围内执行分析，不得返回超出项目、区域或组织权限范围的数据。  
FR13: 系统应向用户展示本次结论是如何产生的，至少包括意图理解结果、计划步骤和关键证据，而不是仅给出黑盒式答案。  
FR14: 系统当前版本只支持物业行业分析问题，对客服系统、客服会话、营销或非分析型业务流程问题不提供产品能力承诺。  
FR15: 在 Growth 阶段，移动端用户可以查看最近一次分析的结论摘要、关键证据和分析状态，但移动端不需要承载完整分析计划编辑能力。  
FR16: 在 Growth 阶段，移动端用户可以提交简短追问或下钻指令，系统应将其并入既有分析会话上下文处理。  
FR17: 系统应将分析执行过程、阶段结果、关键证据和最终结论组织为统一的可渲染内容块，以支持表格、图表、证据卡、执行节点和摘要卡等多种界面呈现方式。  

### 非功能需求

NFR1: 对于复杂分析请求，系统应在用户提交问题后的 5 秒内返回首个可见反馈，包括意图结果、计划生成状态或第一步执行状态。  
NFR2: 对执行时间超过 10 秒的分析任务，系统应持续提供步骤级状态更新，更新间隔不超过 5 秒。  
NFR3: 对所有成功完成的归因分析会话，100% 的结果都必须包含计划步骤摘要和至少一条关键证据说明。  
NFR4: 系统应在 100% 的受保护分析请求中执行权限校验，未经授权的项目、区域或数据主题不得被查询或展示。  
NFR5: 系统应对分析请求、关键工具调用、导出行为和权限失败事件保留审计记录，且审计记录至少保留 180 天。  
NFR6: 系统在试点阶段应达到业务时段 99.5% 的可用性，以平台监控数据为准。  
NFR7: 系统应支持在企业自托管环境中部署，不要求依赖公有云专属托管平台能力才能运行核心功能。  
NFR8: 对同一会话上下文下重复执行的结构化分析请求，系统应在 95% 的情况下保持相同的计划骨架和同类结论排序。  
NFR9: 浏览器端不得直连核心数据源、核心分析服务或模型密钥；所有敏感访问都必须经过服务端受控边界。  
NFR10: 系统应支持从单轮归因分析扩展到多轮循环分析，而不需要重写整体产品交互模型。  
NFR11: MVP 阶段的 PRD、架构和实施故事必须明确区分 PC 后台能力与移动端能力，不得默认将 PC 完整工作流复制为移动端范围。  

### 附加需求

- 采用官方 `create-next-app` 初始化的 Next.js App Router 作为项目起始底座，开发时使用 `next dev` 默认 `Turbopack`。  
- 运行时基线为 `Next.js 16.1`、`React 19.2`、`Node.js 24 Active LTS`。  
- 系统形态为“模块化单体 + 独立后台 Worker”，MVP 阶段不拆分微服务。  
- 当前最新产品方向以 Postgres 作为业务与平台数据存储目标；此变化需要后续与架构文档同步。  
- 语义层采用 `Cube Core 1.6.x`，接入只读副本或分析 schema，不走 ERP 主写路径。  
- 实体关系与因果边能力采用 `Neo4j 5.26 LTS`。  
- 统一校验边界采用 `Zod 4`。  
- 缓存、限流、任务元数据和流式协调采用 `Redis 8.2+`。  
- 认证采用基于 ERP 的服务端会话和带作用域的 RBAC。  
- API 交互模式采用基于 Next Route Handlers 的 `REST + SSE`，MVP 不引入 GraphQL。  
- 前端采用 App Router 服务端优先模式，UI 基础为 `Tailwind CSS 4.1`、`shadcn/ui`，客户端服务端状态同步采用 `TanStack Query v5`。  
- 浏览器端不得直连 Postgres、Cube、Neo4j、Redis 或模型提供方。  
- 部署姿态为自托管容器化，先支持 Docker Compose，再保留向 Kubernetes 演进的边界。  
- 初始进程拓扑至少包含 `web`、`worker`、Postgres、Redis、Neo4j、Cube。  
- 可观测性要求采用 OpenTelemetry、Prometheus、Grafana 和结构化日志。  
- 在引入更多服务拆分前，应优先支持 Worker 独立扩容。  
- `OpenAPI` 仅用于外部或管理面 API；内部契约保持 code-first。  
- `GraphQL`、`gRPC`、专用向量数据库和细粒度多 Agent 拆分延后到 Post-MVP。

## Epic 1: 安全分析工作台与会话入口

用户可以登录到权限范围内的 PC 分析工作台，发起分析、查看历史分析，并在受控范围内使用系统。

### Story 1.1: 初始化 Next.js 分析平台骨架

As a 开发团队,
I want 以官方推荐脚手架建立 Next.js App Router 项目骨架和基础运行脚本,
So that 后续分析工作台、会话和执行能力可以在一致且可演进的工程底座上实现。

**关联需求：** 附加需求（Next.js App Router、Turbopack、Node.js 24、模块化单体起步、官方脚手架）

**Acceptance Criteria:**

**Given** 项目仓库尚未具备正式的 Web 应用骨架
**When** 团队完成首个基础工程初始化
**Then** 仓库中应存在可运行的 Next.js App Router 应用结构、基础脚本和 TypeScript 配置
**And** 开发环境应通过 `next dev` 启动并默认使用 `Turbopack`

**Given** 基础工程初始化完成
**When** 开发者查看项目结构
**Then** 应存在后续可承载 `app`、`domain`、`application`、`infrastructure` 分层的基本目录约定
**And** 项目应基于官方 `create-next-app` 输出进行初始化

### Story 1.2: ERP 身份接入与受保护会话

As a 物业分析用户,
I want 使用现有 ERP 身份进入分析平台,
So that 我不需要重新注册，并且系统能继承我的组织与权限范围。

**关联需求：** FR12；附加需求（ERP 服务端会话、RBAC）

**Acceptance Criteria:**

**Given** 用户已经拥有有效的 ERP 账号
**When** 用户访问分析平台并完成登录
**Then** 系统为该用户创建受保护的应用会话
**And** 会话中包含该用户的组织、项目或区域权限上下文

**Given** 未登录用户访问受保护页面
**When** 系统检测不到有效会话
**Then** 系统应阻止访问受保护分析页面
**And** 引导用户进入登录流程

### Story 1.3: 权限范围内的分析工作台首页

As a 物业分析用户,
I want 进入一个只展示我权限范围内分析入口的工作台首页,
So that 我能从正确的数据范围开始分析，而不会误入不支持或无权限的领域。

**关联需求：** FR12, FR14, NFR11

**Acceptance Criteria:**

**Given** 用户已成功登录
**When** 用户进入分析工作台首页
**Then** 系统应展示分析入口、最近分析入口和当前权限范围提示
**And** 不展示客服系统或其他非物业分析范围的产品入口

**Given** 用户权限仅覆盖部分项目或区域
**When** 用户查看工作台首页
**Then** 系统应只显示其有权访问的分析范围信息
**And** 不暴露超出权限边界的项目或区域名称

### Story 1.4: 创建新的分析会话

As a 物业分析用户,
I want 在工作台中发起一个新的分析会话并输入自然语言问题,
So that 我可以从一个明确的分析上下文开始后续分析流程。

**关联需求：** FR1, FR11

**Acceptance Criteria:**

**Given** 用户已进入工作台首页
**When** 用户点击“新建分析”并输入一个自然语言问题
**Then** 系统应创建一个新的分析会话记录
**And** 保存问题原文、创建时间和用户标识

**Given** 用户提交新的分析问题
**When** 会话创建成功
**Then** 系统应将用户带入该分析会话页
**And** 在后续分析能力尚未执行前保留该问题作为会话起点

### Story 1.5: 历史分析会话列表与回看

As a 物业分析用户,
I want 查看并重新打开我的历史分析会话,
So that 我可以回看之前的问题、结果和后续追问上下文。

**关联需求：** FR11, FR12

**Acceptance Criteria:**

**Given** 用户已存在历史分析会话
**When** 用户进入历史会话列表
**Then** 系统应按时间倒序展示该用户可访问的历史会话
**And** 每条会话至少显示标题摘要、最近更新时间和状态

**Given** 用户点击某条历史会话
**When** 系统加载该会话
**Then** 系统应展示该会话已保存的问题与基础上下文
**And** 不显示其他用户或超出权限范围的会话内容

### Story 1.6: 非支持领域问题的边界提示

As a 物业分析用户,
I want 当我尝试进入非支持领域时收到明确提示,
So that 我知道当前版本只支持物业分析，不会误以为平台支持客服或其他业务系统能力。

**关联需求：** FR14, NFR11

**Acceptance Criteria:**

**Given** 用户输入或选择明显属于客服系统、客服会话、营销或呼叫中心的问题类型
**When** 系统识别该问题超出当前产品范围
**Then** 系统应提示“当前版本仅支持物业分析场景”
**And** 不进入后续分析流程

**Given** 用户停留在分析工作台
**When** 用户查看产品范围说明或帮助信息
**Then** 系统应明确列出当前支持范围与不支持范围
**And** 说明客服系统相关能力不在当前版本范围内

## Epic 2: 基础设施基线与平台持久化

用户可以在稳定、可持续、自托管友好的平台底座上使用分析工作台，系统具备真实持久化、基础缓存和后台执行边界，而不是依赖进程内临时状态。

### Story 2.1: 建立 Docker Compose 本地基础设施基线

As a 开发团队,
I want 通过 Docker Compose 启动 web、Postgres 和 Redis 的本地开发环境,
So that 后续平台持久化和后台执行能力可以在统一且可重复的环境中开发与验证。

**关联需求：** NFR7；附加需求（Docker Compose、自托管容器化）

**Acceptance Criteria:**

**Given** 团队需要在本地开发平台自有基础设施
**When** 开发者启动容器编排
**Then** 应能通过单一 Compose 配置拉起 `web`、`postgres`、`redis`
**And** 本地环境不依赖手工逐个服务启动

**Given** 基线环境已启动
**When** 开发者检查服务状态
**Then** 应存在明确的连接方式、环境变量约定和健康检查入口
**And** 为后续 `worker` 接入保留清晰扩展位

### Story 2.2: 初始化 Postgres 与 Drizzle 平台 schema

As a 开发团队,
I want 建立平台自有 Postgres schema 与 Drizzle 迁移机制,
So that 受保护会话、分析会话和后续结果留存可以脱离内存实现。

**关联需求：** FR11；附加需求（Postgres、Drizzle）

**Acceptance Criteria:**

**Given** 平台需要真实持久化层
**When** 团队初始化数据库访问与迁移流程
**Then** 项目应具备可执行的数据库连接配置与迁移命令
**And** 平台自有 schema 应与外部 ERP 数据边界清晰分离

**Given** 初始 schema 已建立
**When** 团队检查数据库对象
**Then** 应至少存在后续会话持久化所需的最小表结构
**And** 不提前创建超出当前故事范围的大量未来表

### Story 2.3: 将受保护会话持久化迁移到 Postgres

As a 物业分析用户,
I want 我的登录会话保存在真实持久化层中,
So that 系统重启或扩展实例后不会立刻丢失会话状态。

**关联需求：** FR12, NFR4, NFR9

**Acceptance Criteria:**

**Given** 用户已通过 ERP 身份登录
**When** 系统创建受保护会话

## Epic 5: 执行分析并输出归因结论

用户可以让系统按计划执行分析，查看中间结果，并获得带证据的归因结论。

### Story 5.1: 提交分析计划到后台执行

As a 物业分析用户,
I want 将已生成的分析计划提交给系统执行,

**And** 执行任务应与当前分析会话建立明确关联

**Given** 某个步骤需要调用外部数据或分析能力
**When** 系统执行该步骤
**Then** 系统应按计划顺序编排对应能力
**And** 单次分析应支持多步执行而不是只发送一次查询

### Story 5.2: 流式反馈执行进度、稳定事件协议与阶段结果

As a 物业分析用户,
I want 在分析执行期间持续看到当前步骤和阶段结果,
So that 我不需要等到任务全部结束才知道系统正在做什么。

**关联需求：** FR7, FR13, FR17, NFR1, NFR2；附加需求（SSE）

**Acceptance Criteria:**

**Given** 分析任务已开始执行
**When** 任一步骤进入运行、完成或失败状态
**Then** 系统应向会话界面推送最新执行状态
**And** 用户可见状态更新间隔不应超过 5 秒

**Given** 某次分析执行超过 10 秒
**When** 用户保持在会话页面
**Then** 系统应持续展示步骤级进度或阶段性发现
**And** 不应让页面长时间无反馈

**Given** 服务端正在向前端回传执行事件
**When** 客户端消费这些事件
**Then** 系统应输出稳定的 execution event envelope
**And** 客户端应基于统一 render block / message part schema 渲染阶段结果，而不是直接把原始 SSE payload 当作最终 UI 结构

### Story 5.3: 输出带证据与富渲染块的归因结论

As a 物业分析用户,
I want 获得排序后的归因结论和关键证据摘要,
So that 我可以直接理解哪些因素最可能导致指标变化。

**关联需求：** FR8, FR13, FR17, NFR3, NFR8

**Acceptance Criteria:**

**Given** 分析计划已成功执行到可形成结论的阶段
**When** 系统生成归因结果
**Then** 系统应输出排序后的原因列表
**And** 每个原因都应附带至少一条用户可理解的证据摘要

**Given** 同一会话上下文下重复执行结构化归因分析
**When** 输入条件未发生变化
**Then** 系统应尽量保持相同的计划骨架和同类结论排序
**And** 明显偏差应可由证据变化解释

**Given** 系统已经形成最终归因结果
**When** 会话界面渲染结果
**Then** 最终结论应可被渲染为统一结果块
**And** 至少支持原因卡、证据块以及图表或表格中的一种结构化表达

### Story 5.4: 保存步骤结果与最终结论

As a 物业分析用户,
I want 在分析完成后保留计划、主要步骤结果和最终结论,
So that 我之后可以回看、复盘并继续追问。

**关联需求：** FR11, FR13, FR17, NFR3

**Acceptance Criteria:**

**Given** 一次分析执行已完成或中断
**When** 用户稍后重新打开该会话
**Then** 系统应恢复已保存的计划骨架、主要步骤结果和当前结论状态
**And** 用户无需重新执行即可查看上一次主要结果

**Given** 某次分析未能形成最终结论
**When** 会话被保存
**Then** 系统仍应保留已完成步骤和失败位置
**And** 为后续继续执行或复盘提供上下文

**Given** 分析结果需要被 PC 与移动端复用
**When** 系统持久化执行结果
**Then** 持久化对象应包含步骤结果、最终结论块和最小投影字段
**And** 不应只保留纯文本摘要

## Epic 6: 多轮追问、纠偏与重规划

用户可以基于初始结论继续追问、纠正方向、缩小范围，系统支持多轮循环分析。

### Story 6.1: 基于既有结论发起追问

As a 物业分析用户,
I want 在已有归因结论上继续追问更细的问题,
So that 我可以逐层下钻到更具体的原因。

**关联需求：** FR9, FR11

**Acceptance Criteria:**

**Given** 当前会话已经存在一轮归因结论
**When** 用户输入新的追问或下钻问题
**Then** 系统应将该问题附着到原会话上下文中
**And** 不新建脱离上下文的独立分析记录

**Given** 用户追问引用了上一轮中的项目、区域、原因或指标
**When** 系统接收追问
**Then** 系统应默认复用这些已存在的上下文信息
**And** 允许用户再补充新的范围条件

### Story 6.2: 补充因素或缩小分析范围

As a 物业分析用户,
I want 在会话中补充新的候选因素或缩小分析范围,
So that 系统可以围绕我最新关心的方向继续分析。

**关联需求：** FR9, FR10, FR13

**Acceptance Criteria:**

**Given** 当前会话已有既定分析上下文
**When** 用户补充新的因素、限定项目范围或追加比较条件
**Then** 系统应将这些输入并入当前轮次的分析上下文
**And** 在界面中清楚展示哪些条件是后续新增的

**Given** 用户输入的补充条件与现有条件冲突
**When** 系统进行上下文整合
**Then** 系统应提示冲突并要求确认
**And** 不静默覆盖已确认的重要范围条件

### Story 6.3: 根据纠正结果重生成分析计划

As a 物业分析用户,
I want 在发现分析方向偏差后重生成后续计划,
So that 系统的下一轮执行能真正反映我纠正后的分析思路。

**关联需求：** FR10, FR13, NFR10

**Acceptance Criteria:**

**Given** 用户已在当前会话中纠正了目标指标、范围或候选因素
**When** 系统重新规划
**Then** 新计划应基于更新后的上下文生成
**And** 明确标识与上一轮计划相比发生了哪些变化

**Given** 上一轮已有部分结果可复用
**When** 系统生成新计划
**Then** 系统可以复用仍然有效的已有结果
**And** 不应要求整条分析链路无条件从零开始

### Story 6.4: 保留多轮循环历史与结论演化

As a 物业分析用户,
I want 查看不同轮次的计划、结果和结论变化,
So that 我可以理解这次分析是如何逐步收敛到最终原因的。

**关联需求：** FR9, FR11, FR13, NFR10

**Acceptance Criteria:**

**Given** 一个分析会话已经经历两轮及以上分析循环
**When** 用户查看会话历史
**Then** 系统应按轮次展示每轮输入、计划和主要结论
**And** 用户可以区分最新结论与历史结论

**Given** 用户需要复盘某一轮的判断依据
**When** 用户切换到历史轮次
**Then** 系统应展示该轮的关键证据和计划摘要
**And** 不因新一轮结果生成而覆盖历史记录

## Epic 7: 企业级治理、审计与自托管运营

平台管理员和企业方可以在自托管环境中安全部署、审计、限权、监控和运营系统。

### Story 7.1: 服务端权限校验与越权拦截

As a 平台管理员,
I want 所有分析请求都在服务端完成权限校验,
So that 系统不会返回超出用户项目、区域或组织边界的数据。

**关联需求：** FR12, NFR4, NFR9；附加需求（服务端会话、RBAC）

**Acceptance Criteria:**

**Given** 用户已登录并发起一次分析请求
**When** 请求进入服务端分析边界
**Then** 系统应根据当前会话中的作用域权限完成校验
**And** 未授权的数据主题、项目或区域不得进入查询或分析流程

**Given** 用户尝试访问超出权限范围的数据
**When** 权限校验失败
**Then** 系统应返回明确的权限失败结果
**And** 不泄露越权对象的具体敏感数据

### Story 7.2: 记录分析与工具调用审计日志

As a 平台管理员,
I want 对分析请求、关键工具调用和权限失败事件进行审计,
So that 企业可以追溯系统行为并满足内部治理要求。

**关联需求：** NFR5, FR12

**Acceptance Criteria:**

**Given** 用户发起分析、触发关键工具调用或发生权限失败
**When** 事件被系统处理
**Then** 系统应写入可检索的审计记录
**And** 记录至少包含用户标识、时间、事件类型和关联会话

**Given** 平台管理员需要回看历史审计事件
**When** 查询审计记录
**Then** 系统应能提供至少 180 天的保留窗口
**And** 不允许普通分析用户直接查看全量审计明细

### Story 7.3: 建立自托管容器化部署基线

As a 企业运维负责人,
I want 使用标准化容器方式部署平台核心能力,
So that 我们可以在自有环境中运行系统而不依赖公有云专属托管能力。

**关联需求：** NFR7；附加需求（Docker Compose、自托管容器化）

**Acceptance Criteria:**

**Given** 企业需要在自有环境中部署系统
**When** 运维团队按交付说明进行部署
**Then** 核心服务应能够以容器方式启动
**And** 不要求绑定公有云专属托管平台才能运行核心功能

**Given** 平台包含 Web、Worker 和基础设施依赖
**When** 团队查看部署基线
**Then** 应存在能够表达这些组件边界的容器化配置
**And** 为后续向 Kubernetes 演进保留清晰边界

### Story 7.4: 建立运行监控与可用性观测

As a 企业运维负责人,
I want 监控平台的可用性、任务运行状态和关键故障,
So that 我们可以在试点阶段及时发现问题并维持目标可用性。

**关联需求：** NFR6；附加需求（OpenTelemetry、Prometheus、Grafana、结构化日志）

**Acceptance Criteria:**

**Given** 平台已在试点环境运行
**When** 运维团队查看系统监控
**Then** 应能看到基础服务健康状态、分析任务状态和关键错误信息
**And** 这些信息足以支持 99.5% 业务时段可用性的跟踪

**Given** 某个关键组件出现故障或异常
**When** 监控系统捕获异常
**Then** 团队应能从指标、日志或追踪中定位问题范围
**And** 不依赖人工登录多个节点逐个排查

### Story 7.5: 账号/手机号登录与 URL 免密桥接

As a 内部试点用户,
I want 使用账号或手机号加密码登录，并在 URL 携带账号或手机号时直接进入受保护工作台,
So that 我可以复用已同步的 ERP 用户目录与组织权限，快速验证分析链路而不需要注册新账号。

**关联需求：** FR12, NFR4；附加需求（ERP 用户目录、服务端会话、试点登录桥接）

**Acceptance Criteria:**

**Given** 已存在受控同步的用户目录与组织权限关系  
**When** 用户在登录页输入账号或手机号以及密码  
**Then** 系统应基于账号或手机号解析到唯一用户并校验密码  
**And** 成功后创建服务端会话并带入该用户的组织、项目与角色范围  
**And** 不再要求用户手动输入组织、项目、区域或角色信息

**Given** 登录 URL 中包含账号或手机号参数，且内部桥接入口已启用  
**When** 系统能解析到唯一有效用户  
**Then** 系统可直接创建服务端会话并跳转到受保护页面  
**And** 该桥接流程不要求再次输入密码  
**And** 必须明确标记为内部试点或非生产登录路径

**Given** 账号或手机号不存在、命中多条、用户已停用，或密码校验失败  
**When** 系统尝试登录或执行 URL 桥接  
**Then** 系统必须返回稳定错误  
**And** 不创建会话  
**And** 不泄露额外敏感信息

**Given** 用户完成登录  
**When** 系统展示工作台或后续分析入口  
**Then** 该用户的 scope 应来自已同步的用户目录与组织链路  
**And** 权限链路应沿用 `user -> org -> 下级项目组织 -> projectId / ownerId -> 业务事实`

## Epic 8: 移动端结果查看与轻量追问（Growth）

移动端用户可以查看分析结果摘要、接收提醒，并发起简短追问。

### Story 8.1: 移动端查看最近分析摘要与同源结果投影

As a 移动端业务负责人,
I want 在手机上查看最近一次分析的结论摘要和当前状态,
So that 我可以在不打开 PC 工作台的情况下快速了解分析结果。

**关联需求：** FR15, FR11, FR17, NFR11

**Acceptance Criteria:**

**Given** 用户已有可访问的分析会话结果
**When** 用户在移动端打开最近一次分析
**Then** 系统应展示结论摘要、分析状态和最近更新时间
**And** 不要求用户进入完整计划编辑界面

**Given** 用户仅有部分项目或区域权限
**When** 在移动端查看结果摘要
**Then** 系统应只展示其有权查看的会话结果
**And** 移动端能力边界应明显区别于 PC 完整工作台

**Given** 移动端需要展示最近分析摘要
**When** 系统组装结果返回移动端
**Then** 结果应来自与 PC 同源的 render schema 投影
**And** 不应单独维护一套不兼容的摘要协议

### Story 8.2: 移动端查看关键证据

As a 移动端业务负责人,
I want 在手机上看到支持结论的关键证据摘要,
So that 我可以快速判断这次分析结论是否值得继续跟进。

**关联需求：** FR15, FR13

**Acceptance Criteria:**

**Given** 某次分析已经形成归因结论
**When** 用户在移动端查看结果详情
**Then** 系统应展示关键证据摘要和主要原因排序
**And** 信息量应适合移动端快速阅读

**Given** 用户尝试查看复杂计划细节
**When** 当前移动端不支持完整编辑或深度编排
**Then** 系统应限制到只读摘要视图
**And** 明确提示完整分析请使用 PC 工作台

### Story 8.3: 移动端发起轻量追问

As a 移动端业务负责人,
I want 在查看结果后提交一条简短追问或下钻指令,
So that 我可以推动既有分析继续深入而不需要重建会话。

**关联需求：** FR16, FR11, NFR11

**Acceptance Criteria:**

**Given** 用户正在移动端查看一个已有分析会话
**When** 用户提交简短追问或下钻指令
**Then** 系统应将该输入并入原有会话上下文
**And** 不创建脱离原分析记录的新会话

**Given** 追问超出移动端轻量能力边界
**When** 用户尝试输入复杂计划编辑类请求
**Then** 系统应提示该类操作需要在 PC 工作台完成
**And** 仍允许用户提交适合移动端的简短追问
