# Epic 6 Retrospective

Status: done

## Epic Review

### What Went Well

- 多轮追问链路已经从页面状态提升为服务端事实：`follow-up -> merged context -> replan -> execute -> snapshot -> history round`。
- 本轮 review 能抓到真实断口，而不是只停留在 UI 现象。关键问题包括：
  - active follow-up 事实源错位
  - 重规划结果未接到真实执行入口
  - 条件变更后旧计划未失效
  - 多轮历史缺显式轮次关联
- 修复方向整体正确：没有通过降级或兼容去掩盖问题，而是把 follow-up / execution / snapshot 之间的关联补到正式模型里。
- Epic 6 的四张 story 最终都拿到了明确的单文件验证证据，能支撑 `done` 判定。

### What Went Poorly

- Story 级 E2E 测试编排不稳。多个 story 文件都在各自 `test.before` 中执行 `pnpm build`，导致合并运行时频繁撞上 Next build 锁，出现大量假红。
- Sprint 治理顺序不够清晰。Epic 7 当前处于“治理主线未开始，但运行化支线已 done”的状态，说明执行顺序缺少明确约束。
- Epic 6 前半段曾出现“展示看起来对，但真实执行入口没接上”的情况，说明当时对“事实正确”与“显示正确”区分不够严格。

### Key Lessons

- 对这类多轮会话产品，页面展示不能作为唯一事实源。必须优先把关联关系落到服务端和持久化模型。
- 回归测试不能过度依赖整页 HTML 文本负断言。页面结构一扩展，旧断言很容易被误伤。
- Story 级 Web 测试一旦都依赖真实 `next build`，就必须有共享构建和锁控制，否则回归信号不可信。
- Sprint 计划里如果允许“运行化支线插队”，就要在规划工件里显式写出执行顺序和原因，否则状态会看起来混乱。
